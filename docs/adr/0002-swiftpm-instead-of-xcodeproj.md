# ADR-0002: SwiftPM packages + bundle-assembly script instead of .xcodeproj

Date: 2026-06-13 · Status: Accepted

## Context

The Swift shell (app, menu-bar agent, helper) needs a build system. The
conventional choice is an Xcode project; the engineering guide's tree sketch
assumed one.

## Decision

Each Swift component is a **SwiftPM package** (`Tabibu/`, `TabibuMonitor/`,
`TabibuHelper/`) producing an executable; the `.app` bundles (Info.plist,
icons, nesting the monitor as a login item, code signing) are assembled by
**`scripts/make-app.sh`**.

## Rationale

- `project.pbxproj` is a generated, merge-hostile format; hand-authoring one
  without Xcode's GUI is exactly the kind of error-prone guesswork the
  zero-hallucination rule forbids. `Package.swift` is plain Swift: diffable,
  reviewable, scriptable, CI-friendly.
- The Rust static lib links cleanly via SwiftPM `linkerSettings`
  (`-L<root>/build -ltabibu_ffi`), with the header exposed through a tiny C
  target (`CTabibuCore`). Verified working (`--version` prints through FFI).
- Everything (build → bundle → sign → DMG) becomes reproducible shell, which
  the release pipeline (`docs/release.md`) needs anyway.

## Consequences

- No storyboard/asset-catalog conveniences: Info.plist is generated by the
  script; the app icon is generated programmatically (`scripts/make-icon.sh`);
  SVG/PDF resources are loaded at runtime rather than via asset catalogs.
- Xcode still works for development (`open Package.swift`) — debugging and
  SwiftUI previews are unaffected.
- Capabilities that *require* an Xcode-managed target (e.g. a future Endpoint
  Security system extension) would need a dedicated subproject then — noted
  as the trigger to revisit this ADR.
